# Local Database (PGlite + Prisma Client)

## Current approach

The local database is an in-process PostgreSQL (PGlite running in WASM inside the
Tauri webview) accessed through Prisma Client via the `pglite-prisma-adapter`.
Persistence is provided by IndexedDB — the data dir path (from Tauri's
`appLocalDataDir()`) serves as the storage namespace key.

### First-run bootstrap

1. The PGlite instance starts against the persist path.
2. We check for the existence of the `Client` table.
3. If absent, the full DDL from `packages/database/prisma/generated/local-schema.sql`
   (generated at build time by `prisma migrate diff --from-empty --to-schema`) is
   executed against the fresh database.
4. If present, we skip bootstrap entirely.

## Known open question: schema upgrades on existing installations

This setup has no migration mechanism. Applying a schema change (a new model, a
new column, a constraint change) to an already-populated local database on an
existing installation is **unsolved**.

The DDL generated by `prisma migrate diff --from-empty` is only valid against an
empty database. Re-running it against a populated database would fail (tables
already exist) or, worse, silently drop and recreate them.

### Why this is deferred

- The local schema (shared models only) is a subset of the full server schema,
  and neither has changed in a breaking way during this foundational phase.
- PGlite + `prisma-pglite` CLI support can handle dev migrations, but applying
  incremental schema changes to a production Tauri bundle requires a design
  decision that this phase explicitly avoids.

### Options for the future

- **Skip entirely** — the POS desktop is a stateless terminal that re-syncs
  from the server on every fresh install. In this model, the local database is
  ephemeral and a fresh bootstrap on every install is acceptable.
- **`prisma-pglite` CLI wrapper** — intercept `prisma migrate dev` and `reset`
  commands at build time and bundle the resulting migration files, then run
  them against the local PGlite at app startup using the `client.exec()` API
  (similar to how unit-test utilities apply migrations in CI).
- **Custom migration runner** — a simple SQL migration file per change, tracked
  by a `__migration` table in the local database, executed in order on startup.
  The DDL files would be generated at build time via `prisma migrate diff`
  between consecutive schema versions.

Choose whichever of these fits the sync/reconciliation design once it is
specified.
